# AlphaGameExplainer 项目规划与版本路线图

## 一、项目概述

AlphaGameExplainer 是一个国际象棋残局自动解说视频生成工具。

输入：FEN 局面或 PGN 对局  
输出：中文解说文本 → TTS 语音 → 带字幕和棋盘动画的 MP4 视频

核心链路（v1.0 目标）：
```
解析对局 → 表库求解 → 节点压缩 → 子残局追踪 → LLM生成结构化剧本 → 视听效果合成 → 视频输出
                                     ↓                        ↓
                              voiceover + pacing    棋子动画 + 箭头 + 高亮 + TTS + 字幕
```

---

## 二、总体架构设计（四阶段演进）

| 阶段 | 名称 | 目标 |
|---|---|---|
| 一 | 底层真值与状态机 | 引入表库穷举解法、子残局动态追踪、禁止概念约束 |
| 二 | 视频导向中间态重构 | 从"写散文"升级为"写分镜剧本"，输出带画面指令的结构化数据 |
| 三 | 推理引擎升级 | 接入 llama.cpp，Grammar 约束结构化生成，尝试更大模型 |
| 四 | 视听同步与渲染升级 | 画面高亮、战术箭头、动态节奏、关键帧定格 |

## 三、各阶段详细设计

### 阶段一：底层真值与状态机

**核心思想**：把"事实"从 LLM 的职责中剥离，由程序确定，LLM 只负责表达。

**已实现内容**：

1. **本地表库接入**
   - `TablebaseSolver` 封装 Syzygy（5-7子 WDL/DTZ）与 Gaviota（3-5子 DTM）
   - `get_solution()` 优先级：本地表库 > ChessDB > Stockfish + Syzygy 辅助
   - Stockfish 启动时可通过 UCI 选项接入 Syzygy 表库辅助搜索

2. **子残局状态追踪**
   - `describe_endgame()` 对任意材料组合生成中文残局名称
   - `get_forbidden_concepts()` 根据当前局面子力生成禁止概念列表
   - `build()` 为每个压缩节点注入 `sub_endgame_name`、`allowed_concepts`、`forbidden_concepts`、`endgame_changed`
   - 残局类型切换时输出日志

3. **提示词子残局感知**
   - `_build_header()` 将全局"残局类型"降级为"起始残局类型"
   - 移除全局 motifs/mistakes 的硬绑定灌输
   - 每个节点在 prompt 中显示 `【当前残局】`、`【允许概念】`、`【禁止概念】`
   - 规则改为"最后一个节点只总结当前子残局规律"

4. **AnalyzedMove 扩展**
   - 新增 `dtm: Optional[int]` 字段，支持 Gaviota 距杀步数
   - `source` 字段扩展为 `"chessdb" / "sf" / "syzygy" / "gaviota"`

5. **endgame_knowledge 扩展**
   - 新增 `describe_endgame()`：对未匹配知识库的残局自动生成 `"车对象"` 式中文名
   - 新增 `get_forbidden_concepts()`：根据是否有兵/车/象/马/后生成禁止规则

### 阶段二：视频导向的解说 + 视听效果生成

**核心思想**：一个好的讲解视频是多维度的——解说词的质量是基础，但画面表现力、音画同步、
视听节奏同样决定了视频的观看体验。本阶段将 LLM 的职责收敛为「决定关键时刻的视听策略」，
其余全部由程序自动计算。

#### 2.1 优秀象棋讲解视频的视听特征分析

通过调研大量象棋教学视频（Lichess 学习板块、Chess.com 教学、YouTube 创作者内容、
CSDN/知乎上的中文象棋教程），总结出以下关键特征：

**视觉层面**

| 效果 | 作用 | 适用场景 |
|---|---|---|
| 棋子滑动动画 | 展示棋子的实际运动轨迹，而非瞬间跳跃 | 每一步走法都适用，是最基本的动画效果 |
| 起始/目标格高亮 | 让观众一目了然哪颗棋在移动、落到哪里 | 关键步骤必须；非关键步骤可简化 |
| 战术箭头 | 展示控制线、封锁线、攻击方向等抽象概念 | 讲解"盒子法""控制线""对王"等概念时 |
| 多色编码 | 不同含义用不同颜色：红色=威胁/攻击、绿色=安全/优势、蓝色=计划/控制 | 复杂局面需要同时展示多种信息 |
| 关键区域半透明覆盖 | 强调"关键格""控制区域""防线"等区域概念 | 菲利多防线、关键格教学等 |
| 残局类型/阶段标签 | 画面角落显示当前残局名称和阶段 | 残局转换时尤为关键 |

**音频层面**

| 效果 | 作用 | 适用场景 |
|---|---|---|
| 语速变化 | 关键步骤放慢、调整步骤正常 | 减慢让观众消化重要信息；等招/试探可稍快 |
| 停顿 | 给观众反应时间、制造悬念 | 决胜着法前 0.5-1s 定格停顿 |
| 音调变化 | 强调关键术语（如"对王""将杀"） | 概念首次出现或决胜时刻 |
| 背景音（可选） | 低音量白噪音或轻音乐降低听觉疲劳 | 时长 >3 分钟的视频 |

**节奏层面**

| 原则 | 说明 |
|---|---|
| 快慢交替 | 纯等招/调整步骤加快节奏，关键转折点放慢 |
| 画等声 | 绝不能先说完解说再展示画面，或画面变了很久解说还没跟上 |
| 关键帧留白 | 关键转折点（如升变、兑子、将杀）前后各留 0.5s 静帧 |
| 阶段过渡 | 残局类型切换（如车兵对车→车对车）时加入短暂过渡 |

#### 2.2 LLM 负责 vs 程序负责的职责划分

这是整个视频系统设计的最核心原则：**LLM 只决定"哪里需要效果"，程序负责"具体怎么做"。**

| 职责 | 由谁负责 | 原因 |
|---|---|---|
| 解说旁白文本 | **LLM** | 语言表达需要理解和创造力 |
| 每个节点的教学重点 | **LLM + KB** | 知识库提供候选概念，LLM 选择最相关的 |
| 哪些棋子/格子需要高亮 | **程序为主**，LLM 可追加 | 走法的起止格程序已知；LLM 可额外标记关键格 |
| 哪些箭头需要画（方向、颜色） | **程序为主**，LLM 决定是否画 | 控制线、对王关系等由程序从局面计算；LLM 决定种类和颜色 |
| 是否需要停顿/慢放 | **LLM** | "这个时刻重不重要"是叙事判断 |
| 当前节点播放节奏 | **LLM** | `fast/normal/slow/pause` |
| 具体箭头坐标/颜色 RGB | **程序** | 纯几何计算 |
| 棋子移动的中间帧数量 | **程序** | 取决于 fps 和音频时长 |
| TTS 语速/音调 | **程序** | 从 LLM 的 pacing 决策映射到 pyttsx3 参数 |
| 字幕时间对齐 | **程序** | 取决于音频文件实际时长 |

#### 2.3 结构化剧本 Schema 设计

LLM 不输出纯文本，而是输出带视听元数据的结构化 JSON。这是 llama.cpp Grammar 约束的直接目标。

```json
{
  "segments": [
    {
      "id": 1,
      "sub_endgame": "车兵对车",
      "voiceover": "白车退至c1建立菲利多防线——白王守住兵前的关键格e3...",
      "pacing": "slow",
      "visuals": {
        "extra_highlights": ["e3"],
        "arrows": [
          {"from": "a2", "to": "a8", "color": "red", "label": "将军"}
        ],
        "phase_label": "防线成型"
      }
    },
    {
      "id": 2,
      "sub_endgame": "车兵对车",
      "voiceover": "黑王向白兵逼近，试图将白王挤出防线。",
      "pacing": "normal",
      "visuals": {
        "extra_highlights": [],
        "arrows": [],
        "phase_label": ""
      }
    },
    {
      "id": 3,
      "sub_endgame": "车兵对车",
      "voiceover": "黑车突然从侧翼将军！白王被迫离开d线，菲利多防线被正面打破。",
      "pacing": "slow",
      "visuals": {
        "extra_highlights": ["d4"],
        "arrows": [
          {"from": "a1", "to": "d1", "color": "red", "label": "侧翼突破"}
        ],
        "phase_label": ""
      }
    }
  ]
}
```

字段说明：
- `voiceover`（LLM 输出）：该节点的解说旁白，按教学节奏分段
- `pacing`（LLM 输出）：`fast` / `normal` / `slow` / `pause_before`（停顿后播放）/ `pause_after`（播放后停顿）
- `extra_highlights`（LLM 可选输出）：解说中提到的关键格，程序会自动合并起止格
- `arrows`（LLM 可选输出）：战术/战略含义的箭头，程序会自动添加走法箭头
- `phase_label`（程序默认 + LLM 可覆盖）：画面角落的阶段标签

#### 2.4 视频生成的自动化流程（程序职责）

每个节点按以下流程自动生成视频片段：

```
┌─────────────────────────────────────────────────────┐
│ 输入: 节点数据 (走法序列 + LLM 剧本)                    │
├─────────────────────────────────────────────────────┤
│ 1. 解析走法生成帧序列                                   │
│    - 每步棋 3 个关键帧: 走前+箭头 → 走后+高亮 → 干净局面  │
│    - 多着节点做中间帧插值（线性过渡，2-3 中间帧）          │
├─────────────────────────────────────────────────────┤
│ 2. 注入视觉增强                                        │
│    - 自动画走法箭头 (from_sq → to_sq, 黄色)             │
│    - 自动高亮起止格 (黄色/橙色半透明)                    │
│    - 叠加 LLM 指定的额外高亮格 (绿色半透明)               │
│    - 叠加 LLM 指定的战术箭头 (红色/蓝色, 带文字标签)       │
├─────────────────────────────────────────────────────┤
│ 3. TTS 语音合成                                        │
│    - 从 voiceover 生成 WAV                            │
│    - 根据 pacing 调整语速: fast=200, normal=180, slow=150│
│    - 计算实际音频时长, 回填到帧时长分配                   │
├─────────────────────────────────────────────────────┤
│ 4. 帧时长动态分配                                       │
│    - 总时长 = 音频时长                                  │
│    - 走前帧 30% → 走后帧 30% → 干净帧 40%               │
│    - pause_before: 在最前面插入 0.8s 静帧               │
│    - pause_after: 在最后面插入 0.5s 静帧                │
├─────────────────────────────────────────────────────┤
│ 5. 画面叠加                                            │
│    - 左下角: 当前残局类型标签 (第一次出现或切换时加动画)    │
│    - 右下角: 步数计数器                                  │
│    - 顶部: 阶段标签 (当 phase_label 非空时)              │
├─────────────────────────────────────────────────────┤
│ 输出: 该节点的视频片段 (VideoClip) + 字幕条目              │
└─────────────────────────────────────────────────────┘
```

#### 2.5 多子力控制线与概念可视化（程序自动计算）

以下视觉效果不需要 LLM 参与，由程序根据局面自动推断：

| 效果 | 计算方式 |
|---|---|
| 车的控制线 | 从车位向四个方向延伸到棋盘边界或第一个阻挡棋子，半透明红色线条 |
| 后的控制线 | 车线 + 象线合并，橙色线条 |
| 象的斜线控制 | 从象位向四个斜向延伸到边界或阻挡棋子，蓝色线条 |
| 对王关系 | 检测两王在同一行/列且距离=2，绿色虚线连接 |
| 关键格标记 | 对单兵残局，计算兵前方两排的三个关键格，浅绿色高亮 |
| 升变威胁 | 兵在第七排时，升变格用金色闪烁标记 |
| 逼和警告 | 弱方无合法走法且未被将军时，红色闪烁边框 |

这些计算全部在 [board_renderer.py](file:///c:/Users/11487/Desktop/刘逸杰/工作/AlphaGameExplainer/src/board_renderer.py) 中实现，LLM 的 `visuals.arrows` 只是补充性的，用于标注"这个箭头表示什么战术含义"。

#### 2.6 棋盘动画效果实现方案

当前 `render_frames()` 只能生成静态帧，改进为：

**棋子滑动动画**
- 将每步棋拆分为 N 个中间帧（N = fps × 0.3s 约 7 帧）
- 中间帧：棋子位置在起止格之间线性插值
- 吃子时：被吃棋子在第 1 帧就已消失
- 升变时：兵在到达升变格后，下一帧变为后

**高亮渐入渐出**
- 高亮格不突然出现，而是在 5 帧内从 0% 透明度渐变到 100%
- 消除后反向渐出

**箭头动画**
- 箭头从起点向终点"生长"（2-3 帧内完成）
- 关键箭头（如将杀）可加脉冲效果（1 次闪烁）

**实现方式**
- 所有动画帧通过 Pillow 逐帧渲染，不依赖外部动画库
- MoviePy 的 `VideoClip` 支持 `make_frame(t)` 函数，可在任意时间点动态生成帧
- 但考虑到性能，仍建议预渲染所有帧为 PNG，用 `ImageSequenceClip` 合成——这对 8×8 棋盘完全够用

#### 2.7 TTS 语音质量改进

当前 `pyttsx3` + `sapi5` 使用的是 Windows 自带的中文语音，效果机械。

**短期方案（v1.0 使用）**
- 保留 pyttsx3 作为默认引擎，利用 SAPI5 的 `rate` 参数实现语速变化
- 在关键句子之间插入极短的静音段（用 `pydub.AudioSegment.silent()` 生成），模拟停顿
- 调整 `rate` 参数：slow=135, normal=175, fast=210

**中期方案（v1.0+ 可选，无需切换架构）**
- 接入 Edge TTS（`edge-tts` Python 库）：免费在线微软神经网络语音，中文质量远超 SAPI5
- 缺点是需要网络，但生成一次后可缓存 WAV
- 支持 SSML 标记，可实现精确的停顿和语速调整

**长期方案（不考虑，仅记录）**
- 本地神经网络 TTS（如 VITS、Bert-VITS2 的中文模型）需要 GPU 推理，不属于当前项目范围

### 阶段三：推理引擎升级

**核心思想**：利用 llama.cpp 的 Grammar 约束、Prompt Cache、KV Cache 量化，实现更快更稳定的生成。

**待实现内容**：

1. **LLMBackend 抽象接口**
   - 统一 backend 接口：`generate(prompt: str) -> str`
   - 当前 Ollama 实现作为默认 backend
   - llama.cpp 实现作为可选 backend

2. **llama.cpp 后端接入（llama-cpp-python 或 llama-server）**
   - 支持 GGUF 模型直接加载
   - 配置 GPU 层数、上下文长度、量化参数
   - 通过 llama-server 的 OpenAI 兼容 API 调用（最小改动现有代码）

3. **GBNF Grammar 约束**
   - 基于阶段二的剧本 Schema 生成 Grammar
   - 强制 LLM 输出合法 JSON
   - 杜绝"好的，以下是我的解说"等非结构化输出

4. **Prompt Cache 复用**
   - 公共前缀（角色定义、规则）缓存复用
   - 后续 chunk 生成时跳过前缀计算

5. **KV Cache 量化**
   - 16GB 显存下运行 14B 级模型的关键技术

6. **模型选择建议**
   - 优先级：同模型高量化 > 低参数量高量化
   - 推荐尝试 Qwen2.5-14B-Instruct Q4_K_M / Q5_K_M
   - 固定 20 个样本做 A/B 对比基准测试

## 五、技术决策记录

### LLM vs 程序的职责边界
- LLM 只输出"叙事判断"：什么需要强调、用什么节奏、额外标注哪些关键格
- 程序负责所有"机械计算"：坐标、颜色 RGB、帧数、时长分配、几何箭头
- 这一原则在阶段二的 LLM 负责 vs 程序负责表中详细列出

### 为什么选 Syzygy 而非 Nalimov
- Syzygy 与 python-chess 原生集成
- Stockfish 原生支持 SyzygyPath UCI 选项
- WDL + DTZ 双文件设计更适合引擎搜索辅助
- 不需要额外依赖

### 为什么 3-4-5 Syzygy 优先于 6 子
- 现有 6 种知识库残局全部 ≤5 子，3-4-5 100% 覆盖
- 1GB vs 150GB 的存储代价差距悬殊
- 6 子残局的教学价值需要更好的 LLM 能力作为支撑

### 为什么先不做和棋讲解
- 和棋讲解需要完全不同的叙事框架（防守视角 vs 进攻视角）
- 提示词和知识库结构需要较大改造
- 当前阶段优先确保必胜残局的解说质量
- v0.2 先做到"识别 + 明确报错并截断"

### 为什么 llama.cpp 而非 vLLM
- 单机单用户低并发场景，vLLM 的 PagedAttention 优势无法发挥
- llama.cpp 的 GBNF Grammar 约束对这个项目是刚性需求
- llama.cpp 的 Prompt Cache 对多 chunk 共用前缀场景收益显著
- llama.cpp 支持 CPU offload，16GB 显存下弹性更大

### 为什么预渲染帧而非实时 VideoClip.make_frame
- 8×8 棋盘帧数通常不到 1000 帧，预渲染 PNG 总大小 <50MB
- ImageSequenceClip 简单可靠，不需要在 make_frame 中反复加载棋子和字体
- 动画效果在渲染时确定，不需要运行时计算，降低视频合成失败风险
