## 二、问题一：Stockfish 搜索步数拖沓

### 2.1 现象

SF 找到的解法步数偏多，明明 15 步能赢它输出 30+ 步。表库能覆盖的局面（≤5 子）已经通过 Syzygy 解决了，此问题主要出现在：
- 6 子根局面（表库未命中，SF 全权负责）
- 子力衰减到 ≤5 子之前的中间阶段

### 2.2 代码定位

文件：`src/stockfish_analyzer.py`，函数 `_sf_solve`（L100-L262）

当前搜索流程：
```
阶段1: _sf_mate_try (mate priority, 2秒, depth=25)
   ↓ 命中直接返回
阶段2: _sf_step_fast 逐步快速搜索 (0.3~1.0秒/步)
   ↓ 每步贪心选当前最优，不保证全局最短
阶段3: 精选标注 (首3+尾3+将军/吃子步做 heavier search)
```

### 2.3 问题根因

| 问题点 | 代码位置 | 具体说明 |
|--------|---------|---------|
| 逐步搜索缺乏全局视野 | `_sf_step_fast` L49-L63 | 每步仅搜索 0.3~1.0 秒，选"当前看起来最好"的着，不保证整条线是最短将杀路径 |
| 阶段1只在根局面做一次 | L147-L150 | 快速搜索结束后没有再做 mate 验证 |
| 6子根局面不切表库 | L211-L217 | 只有吃掉子且 `piece_count ≤ 5` 且 `tablebase.is_hit()` 才切换，中间每一步都可能是次优的 |
| SF 没有利用 SyzygyProbeLimit | L104-L105 | 代码只设了 `SyzygyPath`，没设 `SyzygyProbeLimit`，SF 搜索时不会主动利用表库信息 |

### 2.4 建议修复

**修复 A（低成本）**：快速搜索结束后增加 mate 验证
```python
# 在 _sf_solve 阶段2结束后，阶段3之前插入：
if fast_moves and not engine_died:
    verify_board = board.copy()
    for am in fast_moves:
        verify_board.push(am.move)
    if not verify_board.is_game_over():
        verify_result = _sf_mate_try(engine, verify_board)
        if verify_result and len(verify_result) > 0:
            # 比较长度，如果 mate 线更短就替换
            # 注意：这里需要从 verify_board 开始验证，而非根局面
            pass
```

**修复 B（中等成本）**：增加 `SyzygyProbeLimit`
```python
# 在 _open_engine() 中：
cfg["SyzygyProbeLimit"] = 6  # 6子及以下局面在搜索中查表库
```

**修复 C（高成本，可延后）**：对关键残局类型（如 KBNvK）做专用搜索策略
- 直接用表库直解（syzygy 文件已有 KBNvK.rtbw/rtbz）
- 不需要走 SF，只需要确认表库命中即可

## 三、问题二：自适应压缩不够

### 3.1 现象

- 17 步的解法压缩力度尚可
- 60 步的解法仍然过于啰嗦，或反之 10 步的解法被过度压缩
- 对用户来说，步数多时解说太长、步数少时太简略

### 3.2 代码定位

文件：`src/storyboard.py`

```python
# L7-L9
MAX_NODE_SPAN = 4        # 正常时每组最多4步
LONG_NODE_SPAN = 6       # 长解法时每组最多6步
LONG_MOVE_THRESHOLD = 18 # ≥18步触发"长解法"模式
COMPACT_NODE_THRESHOLD = 7

# L113-L115
kbnk_mode = endgame_name == "象马杀王" and len(analyzed_moves) >= LONG_MOVE_THRESHOLD
long_line_mode = len(analyzed_moves) >= LONG_MOVE_THRESHOLD
max_span = LONG_NODE_SPAN if long_line_mode else MAX_NODE_SPAN
```

文件：`src/commentator.py`，`_build_json_header` / `_build_chunk_prompt`

- `storyboard.py` 的 `build()` 已经计算了 `compact_mode` 和 `target_length` 字段
- 但 `commentator.py` 的 prompt 构建函数**完全没有使用这两个字段**

### 3.3 问题根因

| 问题点 | 具体说明 |
|--------|---------|
| 压缩阈值是硬开关（18步）| 17 步→少压缩，18 步→多压缩。没有中间地带。60 步和 20 步的压缩力度没有进一步区分 |
| `target_length` 未传给 LLM | `build()` 中已经算出 `"1200-1600字"` vs `"800-1100字"`，但 LLM 不知道 |
| 缺少全局字数预算 | 即使节点标记了 `summary_only`，关键节点仍可能写出超长解说 |
| 节点重要性判断靠标签 | 用"是否将军""是否吃子""eval_delta>200"判断关键性，而非棋理重要性 |

### 3.4 建议修复

**修复 A（低成本，立即执行）**：将 `target_length` 注入 LLM Prompt

在 `_build_json_header` 中：
```python
# storyboard 已计算 target_length
target_length = storyboard.get("target_length", "")
if target_length:
    parts.append(f"【全局字数预算】整段解说总字数控制在{target_length}范围内。")
```

在 `_build_chunk_prompt` 中：
```python
# 为每个节点分配字数预算
node_count = len(chunk_nodes)
total_budget = 1200 if storyboard.get("compact_mode") else 800
per_node_budget = total_budget // max(node_count, 1)
```

**修复 B（中等成本）**：压缩上限改为比例增长

```python
# 替换固定阈值
total_halfmoves = sum(len(cs.sans) for cs in compressed)
# 当总半着数 > 10 时，max_span 从4起，每多5步多1个span，上限10
max_span = min(4 + max(0, total_halfmoves - 10) // 5, 10)
```

**修复 C（高成本，配合第五章）**：切换到语义驱动的关键节点判断——见[第九章](#九自适应压缩的语义驱动改造)。

## 四、问题三：AI 味重、解说空洞

### 4.1 现象

以下是从实际运行中截取的解说词（17 步象马杀王），问题标注见[附录](#十三附录ai-味典型样本与问题标注)。

**核心问题归纳**：

| # | 问题类型 | 具体表现 | 频次 |
|---|---------|---------|------|
| 1 | **套话/比喻重复** | "精妙"出现 7 次、"压缩空间"出现 6 次、"天罗地网/天衣无缝"出现 3 次；"如利剑出鞘"在步8和步11一模一样重复 | 极高 |
| 2 | **缺少重点与原因** | "白方通过马与象的协同推进"——哪匹马？哪个象？走到哪？为什么这样配合有用？完全没有 | 极高 |
| 3 | **浮于表层** | 步2/步9/步12 的解说几乎可以互换——全是"调整子力、压缩空间、配合默契"，没有一句说明每步的实质变化 | 高 |
| 4 | **刻意夸大** | 第一步就说"暗藏杀机"、"如洪水般不可阻挡""耐心的围猎"——非关键步骤被强行说成关键步骤 | 高 |
| 5 | **删去比喻就剩空壳** | 大部分段落删除比喻词后就只剩"白方走了几步，黑王后退了" | 极高 |

### 4.2 代码定位

问题分散在多个文件：

| 文件 | 位置 | 作用 | 问题 |
|------|------|------|------|
| `commentator.py` | L112-L113 `_build_json_header` | "既有技术深度又有叙事感染力"+"浓墨重彩写出张力" | 引导 LLM 进入文学创作模式 |
| `commentator.py` | L151-L152 `_build_json_header` | "禁止棋盘坐标（如h7、g5）" | 禁止了唯一能让解说变具体的路径 |
| `commentator.py` | L210-L248 `_build_chunk_prompt` | 每个节点只给"走法+状态标签+目标短语" | LLM 没有足够信息区分不同步骤 |
| `commentator.py` | L360-L425 `_auto_fix_voiceover` | 自动修正错误叙事 | 只修正了错误，没修正空洞 |
| `storyboard.py` | L659-L809 `build` | 构建节点数据 | 每个节点缺少具体棋理信息 |

### 4.3 根因：禁止坐标 + 缺少棋理信息 = 只能输出空洞比喻

这是一个**结构性矛盾**：

```
LLM 收到的信息:               LLM 被要求做的:            LLM 实际能做的:
"走法: Nb3→Kd6 (2着)          "有技术深度和叙事感染力"    输出:
 状态: 非将军│未将杀│未吃子                             "白方精妙地调整子力位置，
 目标: 继续压缩空间                                     象与马的配合愈发默契，
 详略: 过渡节点                                         压缩对方的生存空间"
                                                       
此外还被禁止说出:             LLM 收到的信息不足以写      LLM 唯一的选择是堆砌比喻
"h7、g5、b3、d4..."          出任何具体分析              来凑"深度"和"感染力"
```

**结论**：AI 味的根本原因不是 LLM 能力不够，而是系统喂给 LLM 的信息太贫瘠——只有状态标签，没有棋理事实。

## 五、根本原因分析：棋理知识断层

### 5.1 三个问题的统一根因

```
                  Stockfish / Syzygy 表库
                         ↓
             能算出"最好走哪步"（WHAT）
                         ↓
              ╔═══════════╧═══════════╗
              ║      ❌ 棋理断层 ❌     ║
              ╚═══════════╤═══════════╝
                         ↓
                    压缩器
              (按将军/吃子标签分组)
                         ↓
                     LLM Prompt
              "走法:X 状态:Y 目标:Z"
                         ↓
                   LLM 输出解说
              "精妙的空间压缩战术"
```

**系统的两个能力源之间有一条断层**：

- **Stockfish/表库**：能精确计算走法、评估值、DTZ/DTM，但输出是机器语言（打分、SAN、UCI）
- **LLM**：能流畅生成中文，但没有任何棋理知识

当前用 Prompt Engineering 和 `endgame_knowledge.py` 来搭桥，但桥的承重不够：

| 手段 | 能做到的 | 做不到的 |
|------|---------|---------|
| endgame_knowledge.py | 告诉 LLM "这是象马杀王，分三个阶段，技法叫W形驱赶" | 告诉 LLM "当前这一步马跳到 d4，控制了 c6 和 e6，黑王从 5 个逃格变 3 个" |
| Prompt 中的claim_level | 告诉 LLM "这个节点可以说将杀""这个节点只能说站位" | 告诉 LLM "这个节点的核心教学点是 X" |
| _auto_fix_voiceover | 替换"双方等待"→"周旋" | 让 LLM 从一开始就不写"双方等待" |

### 5.2 为什么 Prompt Engineering 治标不治本

当前系统依赖的手段可以归纳为"告诉 LLM 不要做什么、要做什么样的风格"，但无法"告诉 LLM 该说什么具体内容"。

**类比**：让一个不懂物理的人讲解物理实验。你给他实验步骤和测量数据，然后说"要有技术深度和叙事感染力"，他只能输出"精妙的引力相互作用在牛顿的框架下绽放出智慧的光芒"——因为你没告诉他引力的具体机制是什么。

### 5.3 解决方向

在 Stockfish/表库和 LLM 之间构建一个新层——**Chess Insight Extractor（棋理洞察提取器）**。这个层用代码（非 LLM）从棋盘状态和引擎信息中提取结构化棋理事实，然后喂给 LLM。

详见[第七章](#七解决方案构建棋理洞察提取层)。

## 六、行业参考方案

### 6.1 C1 论文（ICML 2026）：Master Distillation

**来源**：["Grounded Chess Reasoning in Language Models via Master Distillation"](https://arxiv.org/abs/2603.20510)

**核心思路**：将 Stockfish 的推理过程（Principal Variation）蒸馏为自然语言思维链，再用 SFT+RL 训练小模型。

**关键做法**：
1. 用 Stockfish depth=24 求出 PV（最优走法序列）
2. 将 PV + FEN + 合法走法列表 + 子力信息 作为 Context 喂给一个 frontier LLM（如 Gemini）
3. LLM 通过 "Feigned Discovery Prompting"（模拟发现式提示）生成思维链解释
4. 用引擎评分验证解释的正确性
5. 用这些数据微调 Qwen/较小的模型

**成果**：4B 参数模型达到 48.1% 的象棋谜题准确率，超越其蒸馏老师 Gemini-3-Flash（40.8%），超越所有同规模开源模型。

**对我们的启示**：**LLM 不缺少流畅的语言能力，缺少的是结构化的棋理输入。** 论文的核心贡献不是更好的 prompt，而是找到了将 Stockfish 的输出转化为 LLM 可理解上下文的方法。

### 6.2 Google Gemini Chess Companion

**来源**：[google-gemini/workshops/chess](https://github.com/google-gemini/workshops/blob/main/chess/chess_description_generator.py)

**核心做法**：在 prompt 中注入大量结构化棋理特征：

```
POSITION FEATURES:
- Material: {material_description}           # 子力对比
- King safety: {king_safety_description}      # 王的安全性
- Board control: {board_control_description}  # 棋盘控制
- Pawn structure: {pawn_structure_description} # 兵结构
- Piece activity: {piece_activity_description} # 子力活跃度

COMPUTER ANALYSIS:
- Stockfish evaluation: {evaluation}          # 引擎评分
- Best move: {best_move}                     # 最佳走法
- Alternative moves: {alternatives}           # 候选着
```

**对我们的启示**：这些结构化的棋理特征不需要 LLM 推理，全部可以用 `chess.Board` API + Stockfish 计算出来。把它们组织好喂给 LLM，LLM 就能写出有技术含量的分析。

### 6.3 Caissa-AI

**来源**：[MazenS0liman/Caissa-AI](https://github.com/MazenS0liman/Caissa-AI)

**核心做法**：将棋子位置关系表示为 Neo4j 知识图谱，LLM 通过 LangChain 查询图谱中的结构化棋理信息来生成解说。

**对我们的启示**：知识图谱的思路方向正确但过于重量级。我们不需要全量棋理知识，只需要**针对当前残局类型 + 当前步骤的专项棋理知识**——这正是 `endgame_knowledge.py` 应该承担的角色。

## 七、解决方案：构建棋理洞察提取层

### 7.1 总架构

```
                     ┌─────────────────────┐
                     │   Stockfish 引擎    │
                     │   Syzygy 表库       │
                     │   chess.Board API   │
                     └──────────┬──────────┘
                                │
              ┌─────────────────┼─────────────────┐
              │                 │                 │
              ▼                 ▼                 ▼
       走法 / PV / DTZ    棋盘状态 (FEN)     引擎评估 / Multi-PV
              │                 │                 │
              └─────────────────┼─────────────────┘
                                │
                                ▼
              ┌─────────────────────────────────────┐
              │    ★ Chess Insight Extractor ★      │
              │    (src/insight_extractor.py)       │
              │                                     │
              │  · 空间变化分析                     │
              │  · 关键格/控制格提取                │
              │  · 走法偏差分析 (走错的后果)        │
              │  · 棋理技法匹配 (endgame_knowledge) │
              │  · 阶段/进度判断                    │
              │  · 教学点生成                       │
              └──────────────────┬──────────────────┘
                                 │
                                 ▼
              ┌─────────────────────────────────────┐
              │        压缩器 (storyboard.py)        │
              │  用棋理语义代替标签判断关键节点      │
              └──────────────────┬──────────────────┘
                                 │
                                 ▼
              ┌─────────────────────────────────────┐
              │       LLM Prompt (commentator.py)    │
              │  每个节点含：走法 + 棋理洞察 +       │
              │  教学点 + 空间数据 + 边界条件         │
              └──────────────────┬──────────────────┘
                                 │
                                 ▼
              ┌─────────────────────────────────────┐
              │         LLM 生成解说词               │
              │  有足够信息写出具体、有内容的解说    │
              └─────────────────────────────────────┘
```

### 7.2 洞察提取器的输出结构

对于每个压缩节点，提取器应产出以下结构（全部由 `chess.Board` API / 引擎信息计算，不需要 LLM）：

```python
node_insight = {
    # ===== 1. 走法信息（已有）=====
    "moves": "马b3→d4, 王e5→d6",

    # ===== 2. 空间变化（Board API 直接可算）=====
    "king_legal_before": 5,          # 走之前对方王的合法走法数
    "king_legal_after": 3,           # 走之后对方王的合法走法数
    "space_reduction_pct": 40,       # 活动空间缩减比例
    "pushed_toward": "右下角",       # 对方王被迫移动的方向
    "king_on_edge": False,           # 对方王是否已在边线(1/8排或a/h列)
    "king_in_corner": False,         # 对方王是否已在角落

    # ===== 3. 关键控制格（Board API 直接可算）=====
    "newly_controlled": ["d4", "c6", "e6"],  # 这一着新增了哪些格的控制
    "escapes_cut_off": ["e7", "f7"],         # 对方王的哪些逃格被封堵
    "control_lines": ["第4横排", "c线"],     # 控制线/排/斜线描述

    # ===== 4. 引擎洞察（SF / 表库提供）=====
    "is_only_move": True,            # 这一着是否是唯一保持胜势的着
    "dtz_before": 12,                # DTZ变化
    "dtz_after": 10,
    "dtz_improvement": 2,            # 每步推进的 DTZ 进度
    # 走错的后果（需要 Multi-PV）：
    "deviation_risk": {
        "alt_move": "Nb3→f3",
```

### 7.3 提取器可以计算的具体指标汇总

| 指标 | 计算方法 | 作用 |
|------|---------|------|
| 对方王合法格数量变化 | `len(list(board.legal_moves))` 前后对比 | 量化空间压缩 |
| 对方王被挤的方向 | 对比对方王的位置坐标变化 | 说明推进方向（边线/角落） |
| 新增控制格 | `board.attackers(self_color, sq)` 对比 | 说明子力到位后控制了什么 |
| 逃格被封堵 | 对比走子前后对方王的合法逃逸方向 | 说明战术效果 |
| 王在边线/角落检测 | `square_file(king_sq) in (0,7) or square_rank(king_sq) in (0,7)` | 判断残局推进阶段 |
| 马的同时控制格数 | 检测马所在格能攻击到的对方王逃格方向 | W形驱赶核心指标 |
| 象的同色斜线控制 | 检测象是否在控制与自身同色的斜线，且该斜线在对方王前方 | 象马杀王核心指标 |
| 对王检测 | 两王是否在同一列/排且只差2格 | 对王局面标记 |
| 升变格控制 | 检测兵前方关键格是否被己方王控制 | 兵残局核心 |
| Multi-PV 偏差分析 | `engine.analyse(board, multipv=3)` 对比最优着与次优着的 DTZ/score | 说明"走错会怎样" |

### 9.2 语义重要性的判定规则（由洞察提取器计算）

```python
def compute_importance(node, insight, compressed_stats):
    score = 0
    reasons = []

    # 1. 技法示范（最关键的）
    if insight.get("technique_name"):
        score += 30
        reasons.append(f"演示核心技法：{insight['technique_name']}")

    # 2. 空间变化幅度大
    spatial = insight.get("spatial_change", {})
    reduction = spatial.get("space_reduction_pct", 0)
    if reduction >= 50:
        score += 25
        reasons.append(f"对方王活动格缩减{reduction}%")
    elif reduction >= 25:
        score += 10

    # 3. 唯一着
    if insight.get("is_only_move"):
        score += 20
        reasons.append("这是唯一保持胜势的着，走错会严重退步")

    # 4. 阶段转换
    if insight.get("king_on_edge") and not insight.get("_prev_king_on_edge"):
        score += 20
        reasons.append("对方王被逼到边线——进入新阶段")
    if insight.get("king_in_corner") and not insight.get("_prev_king_in_corner"):
        score += 25
        reasons.append("对方王被逼到角落——收网阶段开始")

    # 5. 对方王逃格归零（即将将杀）
    if spatial.get("king_legal_after", 999) <= 1:
        score += 25
        reasons.append("对方王只剩1格可走，即将将杀")

    # 6. 将军
    if node.get("has_check_in_node"):
        score += 10  # 将军本身不一定重要，但有技法叠加时加强

    if score >= 30:
        return "high", reasons
    elif score >= 15:
        return "medium", reasons
    else:
        return "low", reasons
```

### 10.2 完整的新 Prompt 模板示范

```

- 禁止使用任何比喻和修辞（如"精妙的""天罗地网""如利剑出鞘""暗藏杀机"

过渡节点：写出不超过30字的一句话，说明这一步的目的。

每个节点有 claim_level 控制可用的结论深度：
positioning→只能讲站位和控制
constraining→可以讲空间压缩
forcing→可以讲强制/被迫
terminal→才能说将杀/绝杀

禁止在非terminal节点说将杀；禁止在非将军节点说将军。
禁止使用引擎术语（评估值、DTM、mate in N）。

JSON格式：
{"segments":[{"id":int,"sub_endgame":"string","voiceover":"string","pacing":"string"},...]}
segments数量必须等于本块节点数。
```

### 10.3 过渡节点的强制写法

在 `_build_chunk_prompt` 中，当 `importance == "low"` 时：

```
节点X (过渡节点 - 30字以内):
走法: X → Y
目的: [一句话说明这一步的目的]
格式要求: "{走法概要}，{目的}。" 不超过30字，不使用任何比喻。
```

### 11.2 表库优先

对于 `piece_count <= 5` 的局面，优表库直解而非 SF：

```python
# 在 get_solution 中
if piece_count <= 5 and tablebase_solver is not None:
    tablebase_solver.open()
    if tablebase_solver.is_hit(temp):
        tb_result = tablebase_solver.solve(temp)
        if tb_result:
            # 验证完整性
            test_board = temp.copy()
            for am in tb_result:
                if am.move in test_board.legal_moves:
                    test_board.push(am.move)
            if test_board.is_checkmate():
                Logger.success(f"表库直解: {len(tb_result)}步")
                return tb_result
```
